## Lessons Learned

1. **Priority 1 = Must Have 완전 일치** — 요구사항정의서의 Priority 1 분류가 MoSCoW Must 기준("없으면 서비스 핵심 목적 불가")과 정확히 일치. 별도 재판별 없이 바로 Must로 매핑 가능했음. 향후 요구사항 정의 시 MoSCoW 기준을 명시하면 이 단계가 더 빠름.

2. **Quick Win 식별의 핵심 — 데이터 재활용 여부** — FR-004(집중도 차트), FR-005(손익 차트)가 Quick Win이 된 이유는 FR-003, FR-007 데이터를 그대로 재활용하기 때문. 신규 API 연동이 없으면 차트 추가는 저복잡도. 설계 시 "어떤 데이터를 재활용하는가"를 먼저 확인하는 것이 Quick Win 발굴의 핵심.

3. **AI 분석(FR-020)의 Could → Quick Win 격상** — Priority 2이지만 G-3(전략 개선)의 핵심이고 캐시 설계가 요구사항에 이미 기술되어 있어 복잡도가 M 수준. 단순히 우선순위 숫자만 보지 않고 페르소나 목표와 기술 설계 완성도를 함께 고려해야 함.

4. **비기능 요구사항은 전체 포함** — SR/PR/QR/DR/CR 23건 모두 v1.0에 포함. 비기능은 서비스 운영 전제 조건이므로 이연 불가. 단, QR(품질)은 "완성 수준"이 아닌 "기본 동작 수준"으로 포함하는 것이 현실적.

5. **로그아웃(FR-002) Priority 3의 역설** — 공수가 NextAuth signOut() 한 줄에 불과해 Priority 3임에도 MVP에 포함. 공수 대비 가치를 고려하면 우선순위 숫자보다 실제 구현 비용이 중요.
