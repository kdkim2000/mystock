## Lessons Learned

- **용어 추출 범위**: 요구사항정의서에서 95개 이상 후보가 나왔으나 스킬 정책(50개 우선)으로 핵심만 먼저 정의하는 것이 효율적. 확장 용어는 이후 설계 단계에서 필요할 때 정의해도 충분함.
- **대표 용어 선정 기준 명확화**: "요구사항 원문 표현 우선" 원칙을 적용하면 선정 근거가 명확해져 팀 내 이견이 줄어듦. 다음 프로젝트에도 동일 원칙 적용 권장.
- **재무 용어**: PER·PBR·EPS·BPS·ROE·ROA 등 금융 지표는 영어 약어가 이미 업계 표준이므로 한국어 정식 명칭과 영어 원문을 병기하는 방식이 가독성에 유리함.
- **캐시 관련 용어**: `캐시`, `캐시 TTL`, `갱신`은 이 프로젝트의 핵심 아키텍처 개념이므로 초기에 명확히 정의해 두는 것이 이후 API 설계·시퀀스 다이어그램 작성 시 혼선을 줄여줌.
